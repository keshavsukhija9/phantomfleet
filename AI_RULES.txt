================================================================================
WHAT THE RTX 3050 DOES NOT SOLVE (THE DANGER ZONE)
================================================================================

This is where your hackathon demo is actually at risk.
The heaviest parts of your architecture are CPU-bound, not GPU-bound.
Your 3050 will sit at 0% utilization while these run:


1. THE 3-TIER MONTE CARLO SIMULATOR (Section 12)
──────────────────────────────────────────────────
You want to run 50 (Demo) to 300 (Standard) stochastic simulations per
candidate path.

Unless you are writing custom CUDA kernels or using PyTorch tensors for
the simulation math (which your PRD doesn't specify), this relies entirely
on your CPU and Python's multiprocessing or concurrent.futures.

Python's Global Interpreter Lock (GIL) is going to fight you here.

Fix:
  Use multiprocessing.Pool, NOT threading. Example:
    from multiprocessing import Pool
    with Pool(processes=8) as pool:
        results = pool.map(run_single_simulation, paths)
  This bypasses the GIL entirely and spreads runs across your i7 cores.


2. A* PATH PLANNING + OR-TOOLS (Section 11)
─────────────────────────────────────────────
NetworkX and Google's OR-Tools (solving the Vehicle Routing Problem) are
strictly CPU operations.

If your logistics graph gets too dense or the OR-Tools constraint matrix
gets too complex, your CPU will bottleneck — and the 3050 can't help you
at all.

Fix:
  Keep graph hop depth ≤ 4 as specified. Set a hard 3-second timeout on
  OR-Tools. Fall back to A*-only result if solver hits the limit.

──────────────────────────────────────────────────────
Google's OR-Tools Vehicle Routing Problem (VRP) solver is an absolute beast,

The i7 Reality:
  13th Gen Raptor Lake chips have insane single-thread clock speeds
  (often boosting over 5.0 GHz). Your NetworkX A* graph traversals and
  the OR-Tools constraint matrices will execute so fast that you probably
  won't even need the 3-second timeout you specified in Risk Mitigations
  (Section 31).


3. IT HANDLES THE "AGENTIC" OVERHEAD (LangGraph + FastAPI)
────────────────────────────────────────────────────────────
You are running simultaneously:
  - A 14-node LangGraph state machine
  - A FastAPI backend pushing Server-Sent Events (SSE) every tick
  - An in-memory Pandas dataframe for the world state
  - A Streamlit frontend

Standard CPUs stutter under heavy asynchronous context switching —
handling UI updates while calculating ML features.

The i7 Reality:
  The hybrid architecture of the 13th Gen i7 means your OS will likely
  dump the background FastAPI/Streamlit UI tasks onto the E-cores,
  leaving your P-cores completely dedicated to the heavy LangGraph
  reasoning and path planning.

  Your UI won't freeze when the agent makes a complex decision.

MULTIPROCESSING MONTE CARLO SETUP (copy-paste ready)
================================================================================

  from multiprocessing import Pool, cpu_count

  def run_single_simulation(args):
      path, graph_state, shipment = args
      # ... perturbation logic ...
      return {"sla_met": True, "eta_gain": 3.2}   # example

  def run_monte_carlo(paths, graph_state, shipment, n_runs=50):
      args = [(p, graph_state, shipment) for p in paths for _ in range(n_runs)]
      # Use 75% of cores — leave some for Streamlit/FastAPI
      n_workers = max(1, int(cpu_count() * 0.75))
      with Pool(processes=n_workers) as pool:
          results = pool.map(run_single_simulation, args)
      return results

  # On your i7 13th Gen (16 cores): n_workers = 12
  # 50 simulations × 12 parallel = done in ~0.8 seconds
